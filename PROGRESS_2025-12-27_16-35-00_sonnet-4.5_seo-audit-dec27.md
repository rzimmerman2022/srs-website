## 3. Technical Decisions Made

### Decision 1: Multi-Agentic Architecture (6 Agents)

**Decision:** Deploy 6 specialized agents using SDA Agentic Workflow v1.0

**Rationale:**
- Comprehensive audit requires diverse expertise (research, copywriting, QA, strategy)
- Chain-of-Verification (CoVe) protocol requires independent verification
- Contrarian analysis requires different cognitive approach than optimization

**Trade-offs Accepted:**
- More complex workflow vs single-agent approach
- Higher token usage (~107K tokens) vs simpler analysis
- Longer session duration (3.5 hours) vs quick fixes

**Alternatives Considered:**
- Single Sonnet agent with sequential tasks (faster but less comprehensive)
- Dual agent (research + implementation) without verification (less rigorous)

**Outcome:** Successfully deployed 6 agents, each contributed unique value

### Decision 2: Keep "Southwest Resume Services" in Hero Subtitle

**Decision:** Maintain business name in subtitle despite character count increase

**Rationale:**
- Service clarity > meta tag optimization
- QA agent identified "service clarity lost" as critical gap when name removed
- Visitors need to know what service is offered within 2-3 seconds

**Trade-offs Accepted:**
- Longer subtitle (63 chars with name vs 43 without)
- Trade clarity for potential SEO optimization

**Alternatives Considered:**
- Remove name to optimize character count
- Move name to headline instead of subtitle

**Outcome:** Subtitle now: "Southwest Resume Services - Human-led. Research-backed. Built with you, not generated for you."

### Decision 3: General "Human Team" Language vs. Specific Names

**Decision:** Use "We bring..." instead of "Ryan and Jordyn bring..."

**Rationale:**
- Owner requested "our expertise" and "with our team" (general emphasis)
- Specific names (Ryan + Jordyn) already featured on About page with full bios
- Homepage hero should be benefit-focused, not team-focused

**Trade-offs Accepted:**
- Less personal connection vs clearer value proposition
- May not differentiate as strongly from competitors

**Alternatives Considered:**
- "Ryan and Jordyn personally..." (too specific for hero)
- "Our two-person team..." (awkward phrasing)

**Outcome:** Used collaborative "We/You" language throughout

### Decision 4: Three Strategic Paths (Not Single Prescription)

**Decision:** Present 3 strategic options (A, B, C) instead of prescribing one path

**Rationale:**
- Contrarian agent revealed audit optimizes for volume, not necessarily best for premium positioning
- Owner needs to answer strategic questions before tactical implementation
- Business model determines which optimizations are valuable

**Trade-offs Accepted:**
- Delayed implementation vs immediate action
- Requires owner decision-making vs autonomous execution

**Alternatives Considered:**
- Prescribe Path A (volume optimization) based on audit findings
- Implement all "quick wins" regardless of strategy

**Outcome:** Created decision tree framework in SYNTHESIS document

### Decision 5: Sitemap Verification Method

**Decision:** Test live sitemap via curl instead of checking for static XML file

**Rationale:**
- Next.js generates sitemap dynamically via `app/sitemap.ts`
- Static file check would produce false negative
- Runtime verification confirms actual functionality

**Trade-offs Accepted:**
- Requires dev server running vs static file check
- More complex verification vs simple file existence check

**Alternatives Considered:**
- Search for sitemap.xml file (would fail incorrectly)
- Skip sitemap verification (risky)

**Outcome:** Sitemap verified working, 5th blog post added to generation script

   - Each page exports `metadata` object (Next.js 13+ App Router)
   - Proper OpenGraph and Twitter Card tags already implemented site-wide
   - Schema markup comprehensive (100% coverage on public pages)

3. **Blog System Structure**
   - Posts defined in `lib/blog/posts.ts` as data array
   - Dynamic routes via `app/blog/[slug]/page.tsx`
   - 5 blog posts exist (4 were in sitemap, 1 missing)
   - All posts 1,400-2,100 words (high quality, SEO-optimized)

4. **Location Page Consistency**
   - 4 location pages exist (Phoenix, Scottsdale, Mesa, Tempe)
   - Follow similar template structure
   - All have proper schema markup with geo-specific data
   - Missing Chandler + Gilbert despite homepage claiming service

   - Hero component accepts title/subtitle/description/CTAs as props
   - Easy to update without touching component code
   - Consistent across all pages using Hero

6. **Methodology Page Exists**
   - `/app/methodology/page.tsx` found but not audited initially
   - Contains RAI, Client Truth Principle explainers
   - Strategic asset for differentiation

2. **Blog Post Count Mismatch**
   - 5 posts in `lib/blog/posts.ts`
